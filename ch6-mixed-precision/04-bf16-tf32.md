# 04. BF16 의 장점과 TF32

## 🎯 핵심 질문

- BF16 (1+8+7) 이 FP16 (1+5+10) 과 같은 exponent bit 을 유지하는 이유는?
- 왜 modern LLM (Llama, Mistral, Gemma) 이 모두 BF16 을 기본으로 채택했는가?
- BF16 은 loss scaling 이 **불필요** 한데, 정확히 왜 그런가?
- TF32 (Tensor Float 32, A100+) 가 "transparent" 라 불리는 메커니즘은?
- TF32 의 trade-off (FP32 대비 1–2 ULP 오차) 가 ML 에서 실제로 문제가 되는가?

---

## 🔍 왜 이 내부 구조가 PyTorch 사용자에게 중요한가

Mixed precision training 의 역사를 보면:

1. **2018**: Micikevicius et al. — FP16 + loss scaling (반드시 필요)
2. **2019**: Kalamkar et al. (Google) — BF16 제안 (loss scaling 불필요!)
3. **2021+**: NVIDIA, Google, Meta — BF16 이 standard (FP16 은 거의 deprecated)

따라서 다음을 이해해야 합니다:

1. **BF16 이 왜 더 안전한가** — exponent bit 으로 FP32 와 같은 range 유지
2. **BF16 정확도의 trade-off** — mantissa 7 bit 은 심각하지 않은가?
3. **TF32 의 자동 가속** — A100+ 에서 "그냥 켜면 되는" 이유
4. **Hopper 의 FP8** — 미래의 방향성 (극단적 quantization)

이 문서는 IEEE 754 기반의 수학적 정당화, 실제 training 수렴 곡선, 그리고 현실의 deployment pattern 을 다룹니다.

## 📖 직관적 이해

### 왜 Exponent Bit 이 중요한가?

```
FP16 (1+5+10):
  Exponent bit:  5 bits → 2^5 = 32 values → range ~6.1e-5 ~ 6.5e+4
  Mantissa bit:  10 bits → precision ~3–4 decimal
  Problem: gradient 10^{-5}~10^{-7} 이 range 경계에 위험하게 가까움

BF16 (1+8+7):
  Exponent bit:  8 bits → 2^8 = 256 values → range ~1.2e-38 ~ 3.4e+38 (FP32와 동일!)
  Mantissa bit:  7 bits → precision ~2–3 decimal (손해)
  Advantage: gradient 가 어떤 scale이든 safe → loss scaling 불필요
```

**Trade-off**: Precision 을 약간 포기해서 **robustness 를 획득**.

### BF16 의 정확도 분석

```
Mantissa bits 비교:

FP32:     1.XXXXXXXXXXXXXXXXXXXXXXX (23 bits)
          └─ precision ~ 6–7 decimal

BF16:     1.XXXXXXX (7 bits)
          └─ precision ~ 2–3 decimal
          
FP16:     1.XXXXXXXXXX (10 bits)
          └─ precision ~ 3–4 decimal

관찰:
  - BF16 mantissa 는 FP16 보다 짧음
  - 하지만 exponent range 의 안정성 이득 > mantissa 손실
```

## ✏️ 엄밀한 정의

### 정의 4.1 — BF16 (Brain Float 16)

BF16 은 **Google TPU** 에서 설계된 16-bit 부동소수점:
$$x = (-1)^S \times 1.M \times 2^{E - 127}$$

여기서:
- $S$: 1 bit (sign)
- $E$: 8 bits (exponent, bias 127) — **FP32와 동일**
- $M$: 7 bits (mantissa)

**표현 범위**: $[\, \pm 1.2 \times 10^{-38},\; \pm 3.4 \times 10^{38} \,]$ (FP32와 동일)

### 정의 4.2 — Machine Epsilon 비교

| 타입 | Mantissa bits | $\epsilon_{\text{mach}}$ | Relative precision |
|------|---------------|------------------------|-------------------|
| FP32 | 23 | $2^{-23} \approx 1.2 \times 10^{-7}$ | ~6–7 decimal |
| FP16 | 10 | $2^{-10} \approx 9.8 \times 10^{-4}$ | ~3–4 decimal |
| BF16 | 7 | $2^{-7} \approx 7.8 \times 10^{-3}$ | ~2–3 decimal |

**해석**: BF16 은 relative precision 이 FP32 의 약 1/100. 하지만:
- FP16 언더플로우는 완전한 손실
- BF16 정밀도 손실은 recovery 가능 (gradient accumulation, noise tolerance)

### 정의 4.3 — TF32 (Tensor Float 32)

NVIDIA A100+ 의 하드웨어 기능:

**Computation level**: 내부적으로 FP16-like mantissa (10 bits) 사용
**API level**: FP32 input/output 유지 (transparent)

```
FP32 input:   [sign (1) | exponent (8) | mantissa (23)]
              ↓
Internal:     [sign (1) | exponent (8) | mantissa (10)]  ← 13 bits 손실
              ↓
FP32 output:  [sign (1) | exponent (8) | mantissa (23)]  ← 0-padding
```

**효과**: FP32 대비 1–2 ULP (units in the last place) 오차, 하지만 2–8× 가속

---

## 🔬 정리와 증명

### 정리 4.1 — BF16 Loss Scaling 불필요

Gradient scale $g \in [\, 10^{-20}, \, 10^{20} \,]$ 인 경우, BF16 은:
$$\boxed{\text{모든 } g \text{에 대해 FP16 underflow 없음}}$$

**증명**:
- BF16 normal range: $[\, 1.2 \times 10^{-38}, \, 3.4 \times 10^{38} \,]$
- Typical ML gradient: $[\, 10^{-8}, \, 10^{2} \,]$ (극단적으로 확대해도)
- BF16 range 가 gradient 를 **모두 cover** → underflow 불가능 $\square$

따라서 **loss scaling 은 불필요** (FP16 과의 핵심 차이).

### 정리 4.2 — BF16 정확도의 상한

Mantissa 손실로 인한 최악의 상대오차:
$$\left| \frac{\hat{x} - x}{x} \right| \leq 2^{-8} \approx 3.9 \times 10^{-3}$$

ML training 에서 noise 대비 분석:
- Typical batch noise (SGD): $\sim 10^{-2}$ ~ $10^{-1}$
- BF16 precision loss: $\sim 10^{-3}$

따라서 **BF16 오차는 stochastic noise 보다 훨씬 작음** → 수렴에 영향 무시할 수 있음. $\square$

### 따름 정리 4.3 — BF16 vs FP16 Training Convergence

**정리**: BF16 과 FP16 + loss scaling 은 asymptotically 동일한 convergence rate 를 가진다 (loss scaling parameter 적절히 설정 시).

**정성적 증명**:
1. FP16 + S: gradient underflow 회피 → stable learning
2. BF16: inherently wide range → underflow-free learning
3. 둘 다 final convergence 는 FP32 와 동일 (optimizer 는 unscaled/native FP32 gradient 처리)

**실제 차이**: BF16 은 loss scaling tuning 이 불필요 → 구현 간단.

### 실험 2 — BF16 에서 Loss Scaling 없이 Training

```python
import torch
import torch.nn as nn

model = nn.Sequential(
    nn.Linear(256, 128),
    nn.ReLU(),
    nn.Linear(128, 64),
    nn.ReLU(),
    nn.Linear(64, 10)
).cuda()

optimizer = torch.optim.Adam(model.parameters(), lr=0.001)
criterion = nn.CrossEntropyLoss()

x = torch.randn(256, 256, device='cuda')
y = torch.randint(0, 10, (256,), device='cuda')

print("BF16 without loss scaling:")
for step in range(10):
    optimizer.zero_grad()
    
    # Forward in BF16 (NO autocast, manual casting)
    with torch.autocast(device_type='cuda', dtype=torch.bfloat16):
        logits = model(x)
        loss = criterion(logits, y)
    
    # Backward (NO GradScaler!)
    loss.backward()
    
    # Check gradients
    grad_norm = torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
    optimizer.step()
    
    # Gradient statistics
    grads = [p.grad.abs().mean().item() 
             for p in model.parameters() if p.grad is not None]
    print(f"  Step {step}: loss={loss:.4f}, "
          f"grad_norm={grad_norm:.4f}, "
          f"mean_grad={sum(grads)/len(grads):.2e}")
```

### 실험 3 — TF32 투명 가속

```python
import torch
import torch.nn as nn
import time

# A100 에서만 가능 (또는 compute capability >= 8.0)
model = nn.Linear(4096, 4096).cuda()
x = torch.randn(256, 4096, device='cuda')

criterion = nn.MSELoss()
target = torch.randn(256, 4096, device='cuda')

# TF32 비활성화
torch.backends.cuda.matmul.allow_tf32 = False
torch.backends.cudnn.allow_tf32 = False

t0 = time.time()
for _ in range(100):
    y = model(x)
    loss = criterion(y, target)
    loss.backward()
torch.cuda.synchronize()
time_fp32 = time.time() - t0

# TF32 활성화
torch.backends.cuda.matmul.allow_tf32 = True
torch.backends.cudnn.allow_tf32 = True

torch.cuda.empty_cache()
model = nn.Linear(4096, 4096).cuda()
x = torch.randn(256, 4096, device='cuda')

t0 = time.time()
for _ in range(100):
    y = model(x)
    loss = criterion(y, target)
    loss.backward()
torch.cuda.synchronize()
time_tf32 = time.time() - t0

print(f"FP32 (no TF32): {time_fp32:.3f}s")
print(f"TF32 (enabled): {time_tf32:.3f}s")
print(f"Speedup: {time_fp32 / time_tf32:.2f}x")
```

### 실험 4 — BF16 정확도 분석 (vs FP32)

```python
import torch
import torch.nn as nn
import numpy as np

model = nn.Linear(100, 1)
x = torch.randn(100)
y = torch.randn(1)

# FP32 계산
y_fp32 = model(x.float()).squeeze()
loss_fp32 = (y_fp32 - y).pow(2)

# BF16 계산
y_bf16 = model(x.bfloat16()).squeeze()
loss_bf16 = (y_bf16.float() - y).pow(2)

# 오차 분석
rel_error = (loss_bf16 - loss_fp32).abs() / (loss_fp32.abs() + 1e-10)

print(f"FP32 loss: {loss_fp32.item():.6e}")
print(f"BF16 loss: {loss_bf16.item():.6e}")
print(f"Relative error: {rel_error.item():.2%}")
print(f"ULP difference: ~{2**(-7):.1%}")  # BF16 machine epsilon
```

### 실험 5 — Mixed BF16/FP32 Training

```python
import torch
import torch.nn as nn

model = nn.Sequential(
    nn.Linear(256, 128),
    nn.ReLU(),
    nn.Linear(128, 10)
).cuda()

optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)
criterion = nn.CrossEntropyLoss()

# autocast 로 선택적 BF16 (loss/norm 은 FP32 유지)
x = torch.randn(64, 256, device='cuda')
y = torch.randint(0, 10, (64,), device='cuda')

for step in range(5):
    optimizer.zero_grad()
    
    # BF16 + FP32 mixed
    with torch.autocast(device_type='cuda', dtype=torch.bfloat16):
        logits = model(x)
        loss = criterion(logits, y)
    
    # NO GradScaler 필요!
    loss.backward()
    optimizer.step()
    
    print(f"Step {step}: loss={loss:.4f}")
```

## ⚖️ 가정과 한계

| 가정 | 한계 및 대응 |
|------|-------------|
| BF16 loss scaling 불필요 | Gradient accumulation 수가 매우 많으면 (>1000) underflow 가능 — scaling 고려 |
| TF32 precision loss 무시할 수 있음 | Very sensitive loss (e.g., regularization) 는 FP32 유지 권장 |
| Hardware support (A100+) | 이전 GPU (V100, T4) 에서는 TF32 미지원 |

## 📌 핵심 정리

$$\boxed{\text{BF16} = (1+8+7) \text{ with FP32-range exponent}}$$

| 비교 항목 | FP16 | BF16 | TF32 |
|----------|------|------|------|
| Exponent bits | 5 | 8 | 8 |
| Mantissa bits | 10 | 7 | 10 |
| **Range** | $6.1e-5$ ~ $6.5e+4$ | $1.2e-38$ ~ $3.4e+38$ | $1.2e-38$ ~ $3.4e+38$ |
| **Precision** | 3–4 decimal | 2–3 decimal | 4–5 decimal |
| Loss scaling | **필수** | 불필요 | 불필요 (투명) |
| Modern LLM | ✗ | ✓ | ✓ (가속) |

## 🤔 생각해볼 문제

**문제 1** (기초): BF16 은 왜 FP16 대신 8-bit exponent 를 유지했는가? Mantissa 를 완전히 포기하면서까지?

<details>
<summary>해설</summary>

**역사적 이유**: Google TPU 설계 시 (2019) loss scaling 없이 training 하려는 목표.

**기술적 이유**: 
- FP16 의 narrow range ($6.1e-5$ ~ $6.5e+4$) 는 ML 의 여러 gradient scale 을 cover 못 함
- Exponent 를 확장하면 mantissa 공간 감소 (16 bit 제약)
- **Trade-off**: 정밀도 손실 < 범위 이득

**수학적 정당화**: Mantissa precision 은 $O(10^{-3})$ 손실이지만, loss scaling 의 복잡성 제거와 hyperparameter tuning 감소가 더 큼. $\square$

</details>

**문제 2** (심화): TF32 에서 mantissa 13 bits 를 버릴 때, 어떤 bits 를 선택해서 버리는가? 높은 자리 (high precision) 인가 낮은 자리 (low precision) 인가?

<details>
<summary>해설</summary>

**선택**: **낮은 자리** (least significant bits, LSBs) 버림.

**이유**: 
- Mantissa 는 fractional part 이므로, LSB 를 버리는 것이 상대오차 최소화
- High-order bits 를 버리면 완전히 다른 수 → systematic bias 발생

**구체적으로**:
```
FP32 mantissa: 0.XXXXXXXXXXXXXXXXXXXXXXX (23 bits)
                           ↑ 버림 (13 bits)
TF32 mantissa: 0.XXXXXXXXXX00000000000 (7 bits + 16 zero padding)
```

따라서 **ULP (unit in last place) 오차는 최소** → 절대값이 커지면서의 오차는 상대적으로 작음. $\square$

</details>

**문제 3** (논문 비평): Kalamkar et al. (2019) 가 BF16 을 제안한 이후, 왜 FP16 은 아직도 존재하는가? FP16 의 장점이 남아 있는가?

<details>
<summary>해설</summary>

**FP16 의 현존 이유**:

1. **Memory bandwidth**: FP16 이 BF16 보다 작음 (16 bits but same memory footprint) — 매우 큰 모델에서 약간의 이점

2. **Historical compatibility**: V100, T4 등 older GPU 에서 FP16 지원 역사가 더 김

3. **Specific domains**: Computer vision (CNN) 에서는 FP16 + loss scaling 이 여전히 standard (BF16 은 transformer 에 특화)

4. **Performance on certain ops**: 일부 cuDNN kernel 은 FP16 최적화가 BF16 보다 좋음

**미래 방향**: 극단적 quantization (FP8, INT8) 로 가는 중 (Ch6-05, Hopper 의 FP8). $\square$

</details>
